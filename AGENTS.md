# AGENTS.md

本文件定义 Codex / AI 开发助手在“恐龙侠打BOSS”项目中的协作规则。所有自动化修改、代码生成、文档生成和重构都必须遵守这些约束。

## 项目定位

“恐龙侠打BOSS”是面向中文大学生的 AI 课表排程 + 打怪式学习计划工具。核心体验是把学习目标变成 Boss，把阶段任务和每日计划变成副本，再根据课表空闲时间自动安排。

## 通用开发规则

- 所有面向用户的 UI 文案必须使用中文。
- 默认采用移动端优先设计，桌面端在移动端体验稳定后再增强。
- 不要在前端代码、客户端配置、静态资源或提交历史中写入 OpenAI API Key。
- OpenAI API Key 只能通过服务端环境变量读取。
- AI 输出必须是结构化 JSON，不能依赖自然语言正文作为业务数据。
- AI 返回内容进入业务流程前，必须在服务端做 schema 校验和容错处理。
- 排程算法必须有单元测试，不能只依赖手动验证。
- 课程时间不可移动。课程是硬约束，学习任务只能被安排在课程之外。
- 所有时间在数据库和 API 中保存为 UTC。
- 前端展示时间时，必须按用户 timezone 转换。
- 不要引入与 MVP 无关的复杂架构、过度抽象或大型依赖。

## 产品体验规则

- 用户看到的概念应尽量贴近大学生日常表达，例如“今日副本”“Boss 血量”“战斗结算”“课程时间”“DDL”。
- 游戏化反馈要服务于学习计划，不要让动效、奖励或装饰影响任务执行效率。
- 任务逾期后的提示要鼓励用户继续推进，避免羞辱式文案。
- 移动端首页应优先呈现今天要做什么、还有多久、完成后能造成多少伤害。
- 课程、考试、作业、论文、考证等学习数据要清晰可编辑。

## AI 规则

- AI 拆解目标时必须输出 JSON。
- JSON 至少应包含 Boss、阶段任务、每日副本建议、预计时长、依赖关系、风险提示。
- 服务端必须校验 AI 输出字段、类型、时间范围和任务数量。
- 如果 AI 输出无效，系统应返回可恢复错误，并允许用户重试或手动编辑。
- 不要把用户的完整隐私信息发送给 AI；只发送完成拆解所需的最小信息。

## 排程规则

- 课程时间是硬约束，不能被移动、压缩或覆盖。
- 任务时间是软约束，可因未完成、优先级变化或新目标加入而重排。
- 排程必须考虑截止日期、任务预计时长、用户 timezone、每日可用时间和课程占用。
- 自动重排时，应尽量减少对已确认任务的扰动。
- 未完成任务重排后，Boss 可以轻微回血，但不能造成用户无法恢复的惩罚。
- 排程算法相关逻辑必须具备单元测试，覆盖课程冲突、跨天、截止日期、timezone、任务逾期等场景。

## 数据和时间规则

- 数据库存储时间统一使用 UTC。
- API 入参和出参中的时间字段应明确标注语义，例如 `startAtUtc`、`endAtUtc`、`timezone`。
- 前端只负责展示层 timezone 转换，不应改变后端持久化时间语义。
- 用户 timezone 应在用户设置中保存，默认可由浏览器推断。

## 安全规则

- 不提交 `.env`、密钥、访问令牌或真实用户数据。
- 不在日志中输出 OpenAI API Key、用户隐私数据或完整 AI 请求上下文。
- AI 请求和响应应保留必要的调试信息，但必须避免泄露敏感信息。

## 测试规则

- 排程算法必须写单元测试。
- AI JSON schema 校验必须写单元测试。
- 关键用户闭环应逐步补充端到端测试：创建课程、创建 Boss、生成计划、完成今日副本、查看结算。
- 修复排程、时间或 AI 输出相关 bug 时，必须先补充能复现问题的测试。
